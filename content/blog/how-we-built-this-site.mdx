## Why we built it ourselves instead of outsourcing it

We could have paid another studio to make a nice site. Plenty of businesses do, and there is nothing wrong with that. But the whole point of Roble Media Lab is helping growing businesses use practical AI and automation in a grounded way. Selling that without doing it on our own business first would be hollow.

So we treated our own website as a client project. Same discovery, same build process, same standards we apply to paid work. If these tools genuinely let a small team ship real systems quickly, we should be able to show it, not just say it.

## What runs on this site today

In plain terms, here is what this site does right now:

- A full multi-page website, built to load fast on real Kenyan connections.
- A booking page where you can pick a time and get a calendar invite automatically.
- A contact form that lands with a real person: spam-checked, stored safely, and answered within one business day.
- A click-to-chat WhatsApp button, because in Kenya that is where a lot of first conversations actually happen.
- Privacy-conscious analytics, so we can see what is working without spying on anyone.
- The blog and publishing system this very post sits in.

The tools behind it were mostly ChatGPT for thinking through plans, Claude for building and fixing the actual site, Cloudflare for hosting, and GitHub for keeping versions safe. None of them do the thinking for you. They make the building fast. Deciding what is worth building, and in what order, is still the real work, and it is the part we spend the most time on with clients too.

## The parts that did not work the first time

This is the section most build stories leave out, so it is the one we want to be most honest about.

The first solution was usually not the best solution. Our first booking setup used one tool, then moved to another that fit better. Our first contact form ran on a third-party service, then we replaced it with our own build that asks sharper questions and helps us understand a visitor's problem before we ever talk. That is not indecision. That is what improving a system on evidence looks like, and it is the same loop we run on client work.

Some problems took patience rather than cleverness. There was a stretch where an embedded form and the booking widget simply would not behave inside the page, and the fix came from working through it step by step until it did what it was supposed to. That experience is now built into how we estimate: the tools do not remove the problems. They make the problems solvable.

> AI didn't replace the need to think. It replaced the need to know everything before starting.

That single idea shapes how we scope every project.

## The one shortcut we refused to take

Here is the choice we are proudest of, and it cost nothing but honesty.

When we built the site, we set up the sections a normal agency site has. A place for testimonials. A place for client case studies. A "trusted by" strip. The structure is all there, ready.

They are empty. On purpose.

We could have written a few convincing quotes and nobody would have known. But this business is built on being straight with people about what technology can and cannot do, and that has to start with what we publish about ourselves. So those sections stay empty until they are true. The day a project is complete and the client is happy to be named, they fill up honestly. If we would fake a testimonial, why would you trust anything else we tell you?

## What this means if you run a growing business

If you have been told that AI and modern web tooling are only for technical teams, this build says otherwise. What it takes is a clearly named problem, a small first scope, and the patience to improve when the first attempt is wrong. The tools have become good enough that the limiting factor is no longer engineering headcount. It is knowing what to point the tools at.

That judgement is where most of the value sits, and it is learnable. If you want the longer version of how we choose those opportunities, we wrote it up in [a practical guide to AI for small businesses](/blog/ai-for-smes).
